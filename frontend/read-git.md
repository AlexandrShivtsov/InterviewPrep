# Схема найменувань гілок та рекомендації щодо префіксів
Нижче — компактна і практична схема найменувань гілок, яку можна відразу покласти в README або CONTRIBUTING. Підходить для невеликих та середніх команд. Наприкінці — рекомендації щодо повідомлень коммітів (що писати, коли мерджиш фічу в version-1 і т.д.).

## Основні гілки
* master — стабільний виробничий код (production).
* version-1 або release/1.0 — релізна гілка для поточної версії 1.x (сюди збираються фічі та багфікси перед випуском у master).

### Префікси для робочих гілок (рекомендовані)

* feature/ — нові фічі, додавання функціоналу
*Приклади:*  feature/home, feature/footer, feature/add-auth

* fix/ або bugfix/ — виправлення багів
*Приклади:* fix/404-redirect, bugfix/login-timeout

* hotfix/ — термінові виправлення, які потрібно швидко випустити в master
*Приклади:* hotfix/security-patch, hotfix/critical-db-fix

* chore/ — службові завдання: оновлення залежностей, рефакторинг без зміни поведінки
*Приклади:* chore/deps-update, chore/lint-fix

* release/ — підготовка релізу (іноді використовується замість окремої version-X гілки)
*Приклади:* release/1.1.0, release/1.2

* test/ — експерименти або тимчасові перевірки
Приклади:* test/new-build, test/perf-check
